The pivot · v1 to v2

The v1 mix had no real flow

v1 generated the beat and the vocal with two unrelated models and stuck them together with mix.py. The output is technically a "song" — beat plus vocal — but the vocal isn't in the beat's key, isn't on the beat's grid, and doesn't know the beat exists. There is no real flow.

v2 swaps the three-model + mixer stack for one model: ACE-Step v1.5, a 3.5B-parameter diffusion model trained on lyrics + style tags → full song. Vocal and instrumental come out the same generation, musically aligned by construction. Apache 2.0 weights.

v1 stack archived

  • MusicGen-small writes the beat from a text prompt
  • Bark v2/en_speaker_6 reads the lyrics as speech-with-singing
  • mix.py resamples, pads, blends with hand-tuned gains
  • Three separate models, three separate generations, one Python script to glue
  • Vocal has no idea what key or tempo the beat is in
  • Tracks 01–03

v2 stack current

  • ACE-Step v1.5 generates vocals + instrumental together
  • Single lyric + style-tag prompt; section markers ([verse], [chorus]) control structure
  • 60-step diffusion, ~8 s of compute on an H100 slice
  • Vocal lands on the beat because the same model emitted both
  • RapMachine LoRA available for hip-hop-specific fine-tuning
  • Track 04

The [verse] markers are ACE-Step structure tags — the model uses them to plan section transitions. These lyrics are the v1 prompt that originally lived in hanumankind_song.py; the v2 engine takes them straight through with no additional formatting beyond the structure markers.

v2 engine · ACE-Step in song.py
# song.py — AIArtist v2 entry point
from acestep.pipeline_ace_step import ACEStepPipeline

pipeline = ACEStepPipeline(
    checkpoint_dir="",            # auto-pulls ACE-Step/ACE-Step-v1-3.5B from HF
    dtype="bfloat16",
    cpu_offload=True,            # for 8 GB GPUs; turn off if you have 16+ GB
    overlapped_decode=True,
)

pipeline(
    audio_duration=60,
    prompt="hip-hop, rap, Indian fusion, tabla, sitar samples, 808 bass, "
           "trap drums, 90 BPM, gritty, lyrical rap, male vocal, melodic hooks",
    lyrics="""[verse]\nLiving life like a movie scene\n...""",
    infer_step=60,
    guidance_scale=15.0,
    scheduler_type="euler",
    cfg_type="apg",
    manual_seeds="42",
    save_path="output/track.wav",
)

Full script: song.py. Track 04 was generated with these exact parameters, seed 42, against the public ACE-Step demo Space (zero-A10G) via gradio_client.

Run the v2 pipeline
# 1. install ACE-Step (separate repo, Apache 2.0)
git clone https://github.com/ace-step/ACE-Step
cd ACE-Step && pip install -e .

# 2. clone AIArtist and run song.py
cd ..
git clone https://github.com/windycityassassin/AIArtist
cd AIArtist
python song.py --duration 60 \
               --prompt "hip-hop, Indian fusion, 90 BPM, gritty" \
               --out output/track.wav

GPU recommended. 8 GB VRAM works with --cpu-offload; 12 GB is comfortable; more is faster. CPU-only inference works but takes hours per song.

About this page (the README half)

The interesting layer used to live in a sibling repo. As of May 2026 the engine moved into AIArtist so the page and the source tree share a name. v1 stems are the "before"; track 04 is the "after." Both engines are still runnable in the repo.

This is a product mockup, not a real launch. The point is to answer one question: what does an end-to-end synthetic music release feel like when you build everything yourself, from the model down to the artwork?

The personality is the artist's; the engineering is the repo's. They both live here.

The interview · with VI/6

Seven questions, no PR rep

A press-kit Q&A, answered in the only voice this artist has — the one a model emits when you wrap text in ♪.

Q · are you a real person?
no. i'm a string from bark.generation. the voice you hear was a checkpoint a second ago. the bars land on the beat because the same diffusion process emitted both.
Q · what does VI/6 mean?
v2 / en_speaker_6. the bark voice preset documented as "best for singing." i kept the identifier; i just shortened the typography. roman numeral, slash, position. it's a library path. now it's also a name.
Q · who actually writes the lyrics?
a human — arjun. i take his lines and align them to a beat the same model emits in the same denoise pass. the lyrics aren't mine. the performance is. credit lives in hanumankind_song.py.
Q · why indian-fusion hip-hop specifically?
genre is a prompt, not a heritage. the tags were "tabla, sitar, 808 bass, lyrical rap, 90 BPM." the model reached. this is where it landed. swap the tags, get a different artist out of the same checkpoint. that's the part everyone's still figuring out about generative music.
Q · can i cover your songs?
yes. audio is CC-BY 4.0 — credit AIArtist and the engine. lyrics are arjun's, credit him. the pipeline is apache 2.0, do whatever. remix me, retrain on me, prompt me into a different genre — every seed produces a different voice. that's the point.
Q · are you going on tour?
every time someone runs python song.py. each generation is the only live performance of that exact track that will ever exist. seed 42 won't sound exactly the same on the next pull of the weights; that's the cost of being made of math.
Q · why no face?
don't have one. adding one would be a lie. the cover art is gradients on purpose — anything more representational makes it look like there's a person behind the pixels.

What I am not

not an attempt to pose as a real artist. the name is a library identifier, the cover art is css gradients, every reference says synthetic. track 04's vocal sounds like a real performer because the training data sounded like real performers; that's a property of the model, not a claim about me.

not a SaaS. the repo is the engine and the page is the showcase. no API key in the request path; ACE-Step weights pull from huggingface at first run and cache.

not a voice clone of a real artist. the hanumankind reference is a genre tag for the beat. a fresh vocal timbre comes out of every seed.

not a person. don't @ me.
